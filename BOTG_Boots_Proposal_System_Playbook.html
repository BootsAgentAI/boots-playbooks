The System Overview

The Boots Proposal System is a distributed architecture that converts PDFs into AI-generated proposals in 60 seconds. It uses a three-layer stack: frontend (Vercel), backend worker (Fly.io), and storage/orchestration (Supabase).

What Users See

  1. Click "Generate Brain Proposal" button
  2. Upload PDF with meeting notes
  3. Proposal appears in 60 seconds

What's Happening Behind the Scenes

User clicks button (Vercel) ‚Üì Creates job in Supabase ‚Üì boots-brain (Fly) polls for jobs ‚Üì boots-brain extracts PDF text ‚Üì boots-brain generates proposal with Claude ‚Üì Result stored in Supabase ‚Üì Frontend polls and displays result

This architecture lets you scale horizontally (add more Fly workers) without touching your frontend or database. Each system has one job, and it does it well.

The Control Layer Framework

This is the insight that will save you hours of debugging: Every part of your system is controlled by exactly one thing. When something breaks, you don't need to investigate all systems ‚Äî you need to know which system controls that specific piece.

What's Broken Where It's Controlled How to Fix It
Health check timeout fly.toml Git commit ‚Üí push ‚Üí Fly redeploys
API route timing out on Vercel Source code Git commit ‚Üí push ‚Üí Vercel redeploys
Environment variable missing Vercel/Fly Dashboard Add to dashboard (no Git needed)
Database schema wrong Supabase Dashboard Update in Supabase (or migrations)
Job not being picked up boots-brain code OR Supabase query Check logs ‚Üí fix code ‚Üí git push
Proposal generation too slow Source code (Claude calls) Optimize code ‚Üí git push
‚ö° The Golden Insight

You don't edit files on servers. You don't SSH into Fly to change configs. You don't manually patch things in production. Everything is either in Git (code + config) or the cloud provider's dashboard (secrets + env vars). Know which, and you know where to go.

Decision Tree: Where Do I Make the Change?

Is it a config file? (fly.toml, next.config.js, dockerfile)
‚Üí Git. Make the change, commit, push. Fly/Vercel watches and redeploys.
Is it source code? (app.py, api routes, worker logic)
‚Üí Git. Same as above. Change, commit, push.
Is it an environment variable or API key?
‚Üí Dashboard. Vercel dashboard for frontend secrets, Fly dashboard for worker secrets.
Is it database schema or data?
‚Üí Supabase dashboard or migrations. Not in Git (usually).
3

Deployment Pipeline

Your deployment is automated. When you push to GitHub, this happens:

git push ‚Üì GitHub receives commit ‚Üì Vercel detects change (watches your repo) ‚Üì Vercel builds & deploys frontend ‚Üì [If fly.toml changed] Fly detects change ‚Üì Fly rebuilds Docker image ‚Üì Fly restarts machines with new image

The Ideal Change Workflow

  1. Make change in your editor (e.g., increase timeout in fly.toml)
  2. Test locally if possible (sometimes not necessary for config changes)
  3. Commit: git add fly.toml git commit -m "increase health check timeout to 30s" git push
  4. Wait 2-3 minutes for Fly to rebuild and deploy
  5. Verify with curl: fly status -a boots-brain
‚úì Why This Matters

You don't need to spend hours researching how to update a config. You make the change locally, push it, and let your CI/CD do the heavy lifting. This is the opposite of the token-expensive "how do I update this" spirals.

Health Check Failures (The Recent Issue)

This is what you just experienced: boots-brain starts processing a proposal, but takes longer than the health check timeout allows, so Fly thinks it's dead and kills the machine.

The Problem

17:59:10 - Job arrives: "Drafting proposal (pass 1)..." 17:59:31 - 21 seconds pass (timeout is too short) 17:59:31 - Health check fails - Fly kills the machine - Job never completes

The Fix

Controlled by: fly.toml (Git)

Find the health check config and increase the timeout:

[services.http_checks."http"] interval = "10s" timeout = "5s" # ‚Üê Change this to 30s or 45s # After change: timeout = "30s"

Why This Happened

Your proposal generation takes 20+ seconds (extracting text from PDF, calling Claude, formatting output). The health check is a safety mechanism ‚Äî if your app doesn't respond in time, Fly assumes it's hung and kills it. You needed to tell Fly: "Hey, my app is actually fine, it just takes a while."

‚ö†Ô∏è Don't Ignore Health Checks

Don't just set the timeout to 999s and call it a day. If your app genuinely takes 45 seconds to respond to a health check, that's a problem. The health check timeout should be slightly longer than your slowest legitimate operation, not a band-aid for performance issues.

After You Fix It

  1. Edit fly.toml locally
  2. Commit and push to GitHub
  3. Wait for Fly to redeploy (watch with fly status -a boots-brain)
  4. Test by uploading a PDF and checking logs: fly logs -a boots-brain -n
5

PDF Processing Issues

Your PDF workflow is split between two systems: generation (client-side) and extraction (server-side).

PDF Generation (Client-Side)

This uses the browser's native window.print() API. 100% client-side, no server involved. User clicks "Export PDF", browser renders as PDF, downloads to computer. This part is not your problem.

PDF Extraction (Server-Side)

When boots-brain processes a job, it extracts text from the PDF to feed into Claude. This happens in boots-brain on Fly.io. This is where timeouts and extraction failures happen.

If PDF Extraction Is Failing

Is boots-brain timing out?
‚Üí You already know the fix: increase health check timeout in fly.toml
Is extraction returning bad text?
‚Üí Check the PDF extraction library/code. Problem is likely in how text is being parsed from the PDF. Look at boots-brain source code.
Is the PDF file itself too large?
‚Üí Vercel has a 12MB request size limit. If the PDF is huge, Vercel will reject it before it even gets to boots-brain. Check Vercel logs.
Is the job queue backing up?
‚Üí Jobs are piling up in Supabase because boots-brain isn't processing them fast enough. Either you need more Fly workers, or the extraction is inefficient.
6

Boots-Brain Worker Debugging

boots-brain is a Fly.io worker that polls Supabase for jobs, processes them, and stores results. Most issues can be diagnosed with logs.

Check if boots-brain is running

fly status -a boots-brain # Look for: # - At least one machine in 'started' state # - Health check should show 'passing' # - If health check shows 'warning' or 'failed', that's your problem

View recent logs

fly logs -a boots-brain -n # Look for: # - 'claim_next_brain_proposal' appearing every 3-5 seconds = boots-brain is polling # - 'Drafting proposal' = processing started # - Error messages = something went wrong # - 'Health check failed' = timeout issue

View only errors

fly logs -a boots-brain -n | grep -i error # Shows only lines containing 'error' (case-insensitive)

Watch logs in real-time

fly logs -a boots-brain # This will stream logs as they happen (Ctrl+C to stop)
‚úì Logs Are Your Friend

Don't spend hours theorizing. The logs tell you exactly what's happening. "Drafting proposal..." followed by "Health check failed" = timeout. "claim_next_brain_proposal" every 3s with no "Drafting" = no jobs in queue. Read the logs first, always.

Debugging Strategy

You've already learned this, but let's make it explicit so you never spend 100k+ tokens debugging again.

The Right Way (What You Just Did)

  1. Check system status: fly status -a boots-brain ‚Äî Is it running? Healthy?
  2. Look at the logs: fly logs -a boots-brain -n ‚Äî What's actually happening?
  3. Identify the problem: Health check timed out? Job not picked up? PDF extraction failed?
  4. Know where to fix it: Is it Git or Dashboard?
  5. Make the minimal change: One edit, one commit, done.
  6. Verify with logs: Check logs again to confirm the fix worked.

The Wrong Way (What Burned Your Tokens)

  • Asking Claude to "figure out why the PDF isn't working" without checking logs
  • Pasting screenshots and asking for analysis
  • Waiting in long conversations for Claude to debug something you can see in terminal
  • Adding more context thinking "more input = better output"

The Token-Efficient Debugging Session

Start fresh conversation ‚Üì Run: fly logs -a boots-brain -n | tail -20 ‚Üì Paste ONLY the error (20 lines max) ‚Üì Ask: "What's causing this error?" ‚Üì Claude identifies problem ‚Üì You make the fix (Git or Dashboard) ‚Üì Total tokens spent: 8-15k (not 100k+)
üö® Stop Before You Ask Claude

Before pasting anything to Claude, run the terminal commands first. Get the actual error. If you can't reproduce it, you're asking Claude to debug something you don't even understand yet. You'll both waste time.

üèÜ 10 Golden Rules of Boots Proposal System

1
Every system is controlled by one thing. Health checks ‚Üí fly.toml. API routes ‚Üí source code. Secrets ‚Üí dashboard. Logs ‚Üí terminal. Know the mapping, and you'll never be confused about where to make changes.
2
Check logs before asking Claude. Run fly logs -a boots-brain -n and look for errors. 80% of problems are visible in the logs. You'll solve it faster yourself than waiting for Claude to theorize.
3
Make changes in your editor, commit to Git. Don't SSH into servers. Don't manually edit files on Fly. Edit locally, commit, push. Let your CI/CD do the deployment. This is faster and safer.
4
Health check failures are timeouts. If you see "Health check failed," it means your app took too long to respond. Either optimize the code or increase the timeout in fly.toml. Don't panic about the health check itself.
5
Test changes locally before pushing. For code changes, you can test on your machine. For config changes (like fly.toml), you can't really test locally, but read the file carefully before pushing. One mistake in fly.toml can break everything.
6
Use decision trees, not speculation. When something breaks, use the framework from Section 2: What's broken ‚Üí Which system controls it ‚Üí Where to fix it. This replaces hours of "maybe it's this" with minutes of "it's definitely that."
7
Boots-brain polls Supabase. If no jobs appear in logs, no jobs are in the queue. If boots-brain is processing jobs slowly, either there's a bottleneck in the code, or you need more workers. Check logs to know which.
8
PDF generation is client-side, extraction is server-side. Exported PDFs never touch your servers. Text extraction happens in boots-brain. If PDFs "aren't working," the problem is in boots-brain (Fly), not the PDF export button.
9
Start fresh conversations for debugging. New conversation = clean tokens, no context bloat. Paste only the minimal error, ask your question, get your answer. You'll spend 10-20k tokens instead of 80-100k.
10
Measure twice, code once. Before you edit anything, understand what you're changing and why. Read the file, understand the setting, make the change. You don't get do-overs for production configs, so get it right the first time.
